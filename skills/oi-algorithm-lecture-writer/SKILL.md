# OI 算法讲义写作

## 核心目标

输出适合中小学生信息学竞赛训练的中文知识点讲义。讲义要像课堂材料：先让学生知道“为什么学”，再逐步讲清“是什么、怎么做、为什么对、怎么写代码、哪里容易错”。

代码部分使用简洁竞赛风格：默认 C++14/C++17 可提交代码，偏短变量名、全局数组、直接实现、中文短注释，避免复杂封装。

## 开始前

- 如果用户给出已有讲义目录或相关文件，先抽样阅读同类讲义，继承标题层级、语气、代码缩进、图片引用方式和题型安排。
- 如果用户只给算法名，按通用 OI 讲义结构生成；不要停下来索要细节，除非目标读者、难度或用途会明显改变内容。
- 如果讲义要配套现有图片，使用相对路径，例如 `images/kmp_overview_cn.png`。不要凭空引用不存在的图片；没有图片时可以用文生图、代码绘图、ASCII 图、表格和手动模拟替代。
- 如果需要使用文生图生成讲解图，先检查 `config/image_api.local.json` 是否已经填写 `api_key`、`base_url` 和 `model`；缺失时让用户填写后再调用 API。
- 默认面向 CSP-J/S、GESP、普及到提高入门学生。需要更高阶内容时，先讲基础版，再补“进阶/变形”。

## 讲义结构

优先使用 Markdown。完整知识点讲义一般按下面结构组织，可按算法特点增删：

1. `# 算法名/知识点名`
2. 适合对象、前置知识、讲义约定（如下标从 0 还是 1 开始）
3. 为什么需要这个算法：从朴素做法或生活/竞赛问题引入
4. 核心思想：用一句话概括，再展开解释
5. 基本概念和定义：公式、状态含义、数组含义、节点含义等
6. 手动模拟：给小样例，逐步展示数组、指针、队列、栈、状态或递归变化
7. 核心操作/模板：每个操作先讲思路，再给代码
8. 完整模板代码：能直接复制运行
9. 复杂度分析：表格或简短段落说明时间、空间
10. 经典题型/例题：每题给题意、思路、关键代码或完整代码
11. 易错点：下标、初始化、边界、溢出、重复计数、递归深度等
12. 总结：用 3-6 行口诀式回顾核心公式和使用场景

短复习资料可以压缩为：定义、核心公式、模板代码、复杂度、题型、易错点、总结。

## 讲解风格

- 用中文短句，按“先直觉、再形式化、再代码”的顺序写。
- 每一节只解决一个小问题，标题要具体，例如“3.2 查询路径”比“3.2 说明”更好。
- 多用小规模例子，例如 `n=8`、字符串 `ABABAC`、数组 `2 1 5 3`。
- 对抽象概念使用表格、ASCII 图、代码块、公式和手动模拟。
- 公式要紧跟含义解释，不要只堆数学符号。
- 对学生容易混淆的地方，用“注意”“关键”“为什么”单独点出。
- 讲义语言可以亲切，但不要口水化；保持训练资料的清晰度。
- 不写营销式介绍，不写与算法无关的背景故事。

## 内容深度

### 入门专题

适合前缀和、差分、栈、队列、二分、排序、模拟、枚举等：

- 重点讲使用场景、公式来源、下标边界和模板。
- 至少给一个完整输入输出型模板。
- 题型以直接套模板和简单变形为主。

### 数据结构专题

适合树状数组、线段树、Trie、并查集、堆、单调队列等：

- 先说明暴力做法为什么慢。
- 讲清每个数组或节点维护什么信息。
- 核心操作要拆开讲：建、查、改、合并、下传、回溯等。
- 必须包含手动模拟和完整模板代码。
- 复杂度建议用表格对比暴力/前缀和/树状数组/线段树等方法。

### 算法思想专题

适合贪心、动态规划、搜索、图论、字符串匹配、数学专题等：

- 先定义状态、决策、转移或搜索空间。
- 强调正确性直觉：为什么贪心不会错，为什么状态覆盖完整，为什么 BFS 第一次到达最短。
- DP 讲义必须写清 `dp[i]` 的含义、初始化、转移顺序和答案位置。
- 图论讲义必须写清建图方式、边权含义、适用条件和复杂度。

## 代码要求

- 默认使用 C++14/C++17，以如下模板开头：

```cpp
#include<bits/stdc++.h>
using namespace std;
```

- 已知范围时优先全局数组和常量：

```cpp
const int N = 1e5 + 10;
int a[N], s[N];
```

- 变量名使用竞赛常见短名：`n`、`m`、`q`、`i`、`j`、`l`、`r`、`mid`、`ans`、`cnt`。
- 函数名使用常见写法：`dfs`、`bfs`、`find`、`add`、`build`、`query`、`update`、`pushup`、`pushdown`、`solve`。
- 数组默认 1 下标；字符串算法、STL 边界函数、题目天然 0 下标时明确说明 0 下标。
- 注释只解释关键含义和易错逻辑，使用中文短注释，不逐行翻译语法。
- 模板代码要能独立编译；如果只是片段，要明确写“核心函数”或“关键代码”。
- 注意数据范围可能导致 `long long`，讲义中要提醒并在代码里处理。

## 讲解图生成

只有当图片能明显帮助理解时才生成，例如树状数组管辖区间、线段树递归拆分、KMP 指针跳转、BFS 队列扩展、DP 表格转移。简单公式和短流程优先用表格或 ASCII 图。

本 skill 使用本地配置调用 OpenAI 兼容图片生成接口：

- 配置模板：`config/image_api.local.example.json`
- 本地真实配置：`config/image_api.local.json`
- 生成脚本：`scripts/generate_image_openai.py`

真实配置文件不要提交到 git。若 `config/image_api.local.json` 不存在或字段为空，让用户按模板填入：

```json
{
  "api_key": "填写你的 API Key",
  "base_url": "填写 OpenAI 兼容 base_url，例如 https://api.openai.com/v1",
  "model": "填写图片模型名",
  "size": "1536x1024",
  "quality": "medium",
  "output_format": "png",
  "response_format": "b64_json"
}
```

请求使用 OpenAI Images API 兼容格式：

```text
POST {base_url}/images/generations
Authorization: Bearer {api_key}
Content-Type: application/json
```

```json
{
  "model": "配置中的 model",
  "prompt": "讲解图提示词",
  "size": "1536x1024",
  "quality": "medium",
  "output_format": "png",
  "response_format": "b64_json"
}
```

调用示例：

```bash
python3 scripts/generate_image_openai.py \
  --config config/image_api.local.json \
  --prompt "生成一张中文线段树区间拆分教学图，白底，清晰标注 [1,8]、[1,4]、[5,8]，图片简洁明了，容易理解" \
  --output images/segment_tree_split_cn.png
```

图片提示词要求：

- 使用中文标签，白底或浅色背景，线条清楚。
- 强调算法结构、状态变化、指针移动或数组关系，不做装饰性插画。
- 对教学图写明画面中的关键文字，术语保留英文，例如 `lowbit(i)`、`tree[6] 管辖 [5,6]`。
- 生成后在讲义中用相对路径引用，并检查图片文件确实存在。

## 例题写法

每个例题优先使用下面格式：

````markdown
### 题型：区间求和

**题意：** 给定数组，多次询问 `[l,r]` 的和。

**思路：**
先预处理前缀和 `s[i]`，每次答案为 `s[r] - s[l-1]`。

**关键代码：**
```cpp
...
```

**复杂度：** 预处理 O(n)，每次查询 O(1)。
````

如果是完整题解，使用：题目大意、思路、算法、时间复杂度、参考代码。

## 易错点清单

根据知识点选择加入：

- 下标从 0 还是 1 开始，输出是否要转换。
- `s[0]`、`dp[0]`、根节点、空串、空区间的初始化。
- `mid` 上取中/下取中导致死循环的问题。
- `int` 溢出，区间和、路径长度、方案数要用 `long long`。
- 多组数据要清空数组、邻接表、计数器和标记。
- 递归深度可能爆栈时说明风险。
- 懒标记、回溯标记、访问标记的下传或撤销。
- `lower_bound`/`upper_bound` 返回位置可能等于 `n`。
- 字符串算法中长度和下标的区别。

## 输出检查

完成讲义前快速检查：

- 标题层级连续，整篇结构能从浅到深读下来。
- 每个核心变量都解释了含义。
- 每段代码和前文讲法一致，能独立编译或明确标注为片段。
- 下标约定全篇统一，特殊情况已说明。
- 至少有复杂度和易错点。
- 如果引用图片，路径存在或明确需要用户补图。
